## 1) Escolhe um caminho (um só) 🎯

### A. Builder (dev)

- Objetivo: shippar produto, contrato, integração.
- Começa aqui:
  - Ethereum dev docs: https://ethereum.org/en/developers/docs/
  - Solidity by Example: https://solidity-by-example.org/
  - OpenZeppelin (standards): https://docs.openzeppelin.com/

### B. Founder / Operator

- Objetivo: distribuir, vender, operar, crescer.
- Começa aqui:
  - Lean startup kit: https://www.strategyzer.com/
  - Growth loops (visão geral): https://www.reforge.com/
  - Product discovery: https://www.nngroup.com/articles/product-discovery/

### C. Research / Analyst

- Objetivo: análise, pesquisa, narrativa, teses.
- Começa aqui:
  - Messari (market intel): https://messari.io/
  - Dune (dashboards): https://dune.com/
  - Token Terminal: https://tokenterminal.com/

## 3) Checklist (30 dias, sem drama)

### Semana 1: Base

- [ ] escolhe 1 caminho (dev / founder / research)
- [ ] define 1 micro-objetivo (ex: “publicar um MVP”)
- [ ] cria repo + deploy vazio (hello world)

### Semana 2: MVP

- **Dev:** endpoint + UI + uma feature real
- **Founder:** landing + waitlist + 1 oferta clara
- **Research:** 1 tese + 1 dashboard + 1 post público

### Semana 3: Feedback

- [ ] 10 pessoas testam (não 2)
- [ ] escreve as falhas (não as opiniões)
- [ ] corrige 3 coisas, ignora 20

### Semana 4: Ship + contar a história

- [ ] “release notes” (o que mudou)
- [ ] 1 caso real (antes/depois)
- [ ] partilha o link em 3 lugares

## 4) Exemplo de estrutura de artigo (fácil pro cliente)

Se o cliente não é editor, dá-lhe uma estrutura repetível:

- **Problema**
- **O que tentámos**
- **O que funcionou**
- **O que não funcionou**
- **Links**
- **Próximo passo**
